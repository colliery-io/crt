# Consolidate PTY spawn thread-spawning duplication

## Parent Initiative

[[CRT-I-0030]] — Readability and Module Decomposition

## Objective

Extract a shared `spawn_pty_threads()` helper from the ~80% overlapping code in `spawn_with_options()` and `spawn_with_cwd()` in `crates/crt-core/src/pty.rs`, eliminating ~130 lines of duplication.

## Acceptance Criteria

## Acceptance Criteria

## Acceptance Criteria

- [ ] New private `fn spawn_pty_threads(pair: PtyPair, child: Box<dyn Child + Send + Sync>) -> anyhow::Result<Pty>` helper extracted
- [ ] `spawn_with_options()` calls the helper after its environment/init-script setup
- [ ] `spawn_with_cwd()` calls the helper after its environment setup
- [ ] Reader thread, writer thread, and channel setup code exists in exactly one place
- [ ] Existing `spawn_shell` unit test and all shell integration tests pass unchanged
- [ ] Net line reduction of ~100+ lines in pty.rs

## Implementation Notes

### Technical Approach
1. Identify the shared code block: channel creation, reader thread spawn, writer thread spawn, Pty struct construction (currently duplicated in both spawn functions)
2. Extract into `fn spawn_pty_threads(pair: PtyPair, child: Box<dyn Child + Send + Sync>) -> anyhow::Result<Pty>`
3. Each spawn function retains only its unique setup: `spawn_with_options` keeps shell detection, semantic prompt args, ZDOTDIR setup; `spawn_with_cwd` keeps its simpler env setup
4. Both call `spawn_pty_threads(pair, child)` as their final step

### Dependencies
None — self-contained in `crates/crt-core/src/pty.rs`.

### Risk Considerations
- **Low risk:** The shared code is structurally identical between both functions. The refactoring is mechanical extraction with no logic changes.
