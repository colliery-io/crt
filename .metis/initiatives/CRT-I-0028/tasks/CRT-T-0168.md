# Extract InputContext and split keyboard handler into pure + applied

**Phase 3 — Input handler refactoring. Independent of Phase 1 and Phase 2.**

## Parent Initiative

[[CRT-I-0028]]

## Objective

Split `handle_keyboard_input()` in `src/input/keyboard.rs` into a pure decision function and a side-effectful application function. Create an `InputContext` struct that captures the minimal read-only state needed for keyboard action determination, enabling unit testing without constructing a full `WindowState`.

## Implementation Notes

### Current Problem

`handle_keyboard_input()` (`src/input/keyboard.rs:42-47`) takes `&mut WindowState` and both:
1. Reads state to decide what action to take (pure logic)
2. Mutates state as a side effect (sets dirty flag, calls request_redraw, modifies selection, etc.)

This means testing "does Cmd+T produce NewTab action?" requires constructing a full WindowState with GPU resources, tab bar, shells, etc.

### InputContext Design

```rust
pub struct InputContext {
    pub active_tab: Option<TabId>,
    pub tab_count: usize,
    pub terminal_mode: TermMode,
    pub has_selection: bool,
    pub search_active: bool,
    pub rename_active: bool,
    pub context_menu_visible: bool,
    pub bracketed_paste: bool,
    pub is_fullscreen: bool,
}
```

### Challenges

The current keyboard handler calls several helper functions that also take `&mut WindowState`:
- `handle_scroll_shortcuts(state, ...)` 
- `handle_shell_input(state, ...)`
- `handle_tab_editing(state, ...)`
- `paste_to_terminal(state, ...)`
- `clear_terminal_selection(state, ...)`

These helpers mix decision and application. For the pure split, the decision function needs to determine the _intent_ (e.g., "paste clipboard contents") and the application function performs it. Some helpers may need to be partially decomposed.

### Pragmatic Approach

Start by extracting the top-level action determination (which shortcut was pressed → which action). The deeper helpers (paste, selection, shell input) can remain as side-effectful operations called from `apply_keyboard_action()`. This gives us testable shortcut mapping without requiring a full decomposition of every helper.
