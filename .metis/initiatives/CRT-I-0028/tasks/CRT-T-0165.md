# Create PtyBackend trait and implement for Pty

**Phase 2 — PTY abstraction. No dependencies on Phase 1 tasks. Can run in parallel.**

## Parent Initiative

[[CRT-I-0028]]

## Objective

Define a `PtyBackend` trait in `crates/crt-core/src/pty.rs` that abstracts the PTY I/O interface. Implement the trait for the existing `Pty` struct. This is the foundation for CRT-T-0166 (MockPty) and enables deterministic terminal testing.

- [ ] `Pty` struct implements `PtyBackend`

## Implementation Notes

### Proposed Trait

```rust
pub trait PtyBackend: Send {
    /// Write data to the PTY (keyboard input)
    fn write(&self, data: &[u8]);
    /// Try to read available output from the PTY (non-blocking)
    fn try_read(&self) -> Option<Vec<u8>>;
    /// Read all available output from the PTY (non-blocking)
    fn read_available(&self) -> Vec<u8>;
    /// Resize the PTY
    fn resize(&self, cols: u16, rows: u16);
    /// Shutdown the PTY
    fn shutdown(&self);
    /// Get the process ID of the shell
    fn process_id(&self) -> Option<u32>;
    /// Get the current working directory of the shell process
    fn working_directory(&self) -> Option<PathBuf>;
}
```

### Implementation for Pty

The `Pty` struct already has all these methods as inherent methods. The trait implementation is a direct delegation — each trait method just calls `self.method_name()`. This can be done by either:

1. Moving the inherent impls to the trait impl (preferred — avoids duplication)
2. Adding a trait impl that delegates to inherent methods

Option 1 is cleaner but changes the call site from `pty.write(data)` to the same thing (trait methods are called identically when the concrete type is known).

### Risk Considerations

Low risk — purely additive. The trait doesn't change any existing behavior. The only consideration is whether `Send` bound is sufficient or if `Sync` is also needed (check how Pty is used across threads).
