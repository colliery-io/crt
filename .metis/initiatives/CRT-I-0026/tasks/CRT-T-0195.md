# Drag initiation and threshold detection

## Objective

Wire drag initiation into mouse handling so that clicking a tab starts a potential drag (instead of immediately selecting), and a small mouse movement beyond a threshold activates the drag. If the mouse is released without exceeding the threshold, treat it as a normal tab click/select.

### Key Behavior
- **Last-tab guard**: If the window has only 1 tab, never initiate a drag (tab click behaves as today).
- **Drag threshold**: ~5px movement before drag activates (prevents accidental drags from imprecise clicks).
- **Click-through**: If mouse released before threshold, select the tab normally.
- **Close button**: Clicks on the close button area should never start a drag.

## Implementation Notes

### Current flow (src/input/mouse.rs:311-317)
```rust
ElementState::Pressed => {
    if !handle_terminal_mouse_button(state, x, y, ..., btn, true)
        && btn == MOUSE_BUTTON_LEFT
    {
        handle_tab_click(state, x, y, Instant::now());
    }
}
```

### New flow
On press: if `tab_bar.hit_test(x, y)` returns `Some((tab_id, false))` AND tab_count > 1, create `TabDragState` with `drag_active: false`. Don't select the tab yet.

On move: if `drag_state` exists and not yet active, check distance from start. If > threshold, set `drag_active = true`.

On release: if `drag_state` exists but not active → normal click (select tab, handle double-click for rename). If active → execute drop action (handled by later tasks; for now just cancel).

### Plumbing challenge
`handle_mouse_input` takes `&mut WindowState` but drag state lives on `App`. Options:
1. Pass `&mut App` to mouse handler (big signature change)
2. Return a `MouseAction` enum from mouse handler, let `App` interpret it
3. Store drag state on `WindowState` temporarily, move to `App` when drag crosses window boundary

**Recommended**: Option 2 — keep mouse handler pure-ish, return an action enum. This matches the existing `determine_click_target` pattern.

### Files to modify
- `src/input/mouse.rs` — modify `handle_mouse_input` and `handle_cursor_moved`
- `src/input/mod.rs` — modify `handle_tab_click` to not fire during potential drag
- `src/app/handler.rs` — interpret mouse actions and manage `App.drag_state`

## Status Updates

### 2026-03-26: Implementation complete

**Approach chosen:** Direct App-level interception in `handler.rs` rather than the action-enum approach from the original plan. This was simpler — the handler already has access to both `self.drag_state` and `state` (via `self.windows.get_mut`), so we intercept left-button press/release events before they reach `handle_mouse_input`.

**Changes made:**

1. **`src/input/drag.rs`**: Added `DRAG_THRESHOLD` constant (5px), `exceeds_threshold()` method on `TabDragState`, and `should_start_drag()` guard function that checks: not editing, not context menu, >1 tab, hit test is non-close-button tab.

2. **`src/app/handler.rs`**:
   - **CursorMoved**: Updates `drag_state.current_pos` and checks threshold. Sets `drag_active = true` when exceeded.
   - **MouseInput Pressed (left)**: Calls `should_start_drag()`. If it returns a tab_id, creates `TabDragState` and skips normal mouse handling.
   - **MouseInput Released (left)**: If drag was pending (threshold not exceeded), calls `handle_tab_click` for normal behavior (select/double-click-rename). If drag was active, logs and cancels (drop behavior for later tasks).
   - **KeyboardInput Escape**: Cancels active drag before reaching keyboard handler.

3. **Unit tests (7 new)**: `new_drag_state_is_pending_and_inactive`, threshold tests (stationary, small move, large move, diagonal, boundary).
