# Tab detachment to new window

## Objective

Implement the core detach operation: extract a tab and its shell from the source window, create a new window at the cursor position, and transfer the tab+shell into it. The PTY session must continue uninterrupted.

### User interaction
1. User drags a tab outside all windows (drop target = Detach)
2. User releases mouse
3. A new window appears at the cursor position with the dragged tab
4. The shell session continues without interruption

- [ ] `content_hashes` entry moves with the tab

## Implementation Notes

### Tab extraction (crates/crt-renderer/src/tab_bar/state.rs)

```rust
pub fn remove_tab(&mut self, id: u64) -> Option<Tab> {
    let idx = self.tabs.iter().position(|t| t.id == id)?;
    let tab = self.tabs.remove(idx);
    // Adjust active_tab index
    if self.active_tab >= self.tabs.len() && !self.tabs.is_empty() {
        self.active_tab = self.tabs.len() - 1;
    }
    Some(tab)
}
```

### Lightweight window creation

`App::create_window()` (src/app/initialization.rs) currently:
1. Creates window + GPU state (must keep)
2. Creates a new `ShellTerminal` (skip for detach)
3. Creates initial tab in `TabBarState` (skip for detach)

Options:
- **A**: Refactor `create_window` to accept an `Option<(Tab, ShellTerminal)>` — if Some, use it instead of creating new
- **B**: Extract GPU setup into a helper, create two entry points: `create_window()` and `create_window_for_detach()`

**Recommended**: Option A — less code duplication, and the conditional is clean.

### Shell resize on transfer

The new window may have different dimensions than the source. After inserting the shell, call `shell.resize(Size::new(new_cols, new_rows))` to match the new window.

### Window positioning

Use `winit::window::WindowAttributes::with_position()` to place the new window at the cursor's screen position. The cursor position during drag is in screen coords (from drop target resolver).

### Dependencies
Blocked by: CRT-T-0197 (needs drop target resolver to know when we're in Detach mode)

### Risk: PTY stability
The PTY uses channels (`Sender<PtyInput>`, `Receiver<Vec<u8>>`). Moving the `ShellTerminal` struct between HashMaps doesn't affect channels — they're owned by the struct, not the HashMap. The background reader/writer threads keep running. This should be safe but needs manual verification.
