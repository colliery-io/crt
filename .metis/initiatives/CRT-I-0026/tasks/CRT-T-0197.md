# Cross-window drop target resolution

## Objective

Build the drop target resolver: a pure function that, given the cursor's screen-space position and all window rects + tab bar regions, determines what will happen if the user drops the tab right now. This is the decision brain for the entire drag system — it outputs a `DragDropTarget`.

### Resolution logic
1. Convert cursor from window-local to screen coordinates
2. Check each window (excluding source) for tab bar hit → `Merge`
3. Check source window's tab bar → `Reorder`
4. If cursor is over a window but not its tab bar → `Detach` (dragged off tab bar but still over window body is treated as detach intent — matches browser behavior)
5. If cursor is outside all windows → `Detach`

## Implementation Notes

### Screen coordinate conversion

winit provides `Window::inner_position()` (or `outer_position()`) to get window position on screen. During a drag, the app needs to query all window positions to build the resolver input.

```rust
pub struct WindowScreenRect {
    pub window_id: WindowId,
    pub origin: PhysicalPosition<i32>,  // screen position
    pub size: PhysicalSize<u32>,
    pub tab_bar_height: f32,  // physical pixels
}
```

### Tab bar hit region
The tab bar occupies `y: 0..tab_bar_height` in window-local coords. For merge, we also need the tab rects within that bar to compute insertion index (reuse `compute_reorder_index` from CRT-T-0196).

### Collecting window rects
During a drag, `App` iterates `self.windows` to build a `Vec<WindowScreenRect>`. This happens on every mouse move during drag — should be cheap (iterate HashMap, query positions).

### Files to create/modify
- `src/input/drag.rs` — add `resolve_drop_target()`, `WindowScreenRect`
- `src/app/handler.rs` — collect window rects and call resolver during drag move events

2. **`src/app/handler.rs`**: Pre-collects `Vec<WindowScreenRect>` from all windows before the mutable state borrow (avoids borrow conflict). During CursorMoved, converts cursor to screen coords and calls `resolve_drop_target` to update `drag_state.drop_target`.

**Borrow challenge**: `state` mutably borrows one entry from `self.windows`, but we need to read all windows for rect collection. Solved by collecting rects *before* the `get_mut` call, only when a drag is active (lazy — no cost when not dragging).

**Test limitation**: `WindowId::dummy()` always returns the same value, so merge-path unit test can't distinguish source vs target. The merge logic is structurally the same as reorder with a source_id check — verified 4 other drop target tests (outside→Detach, source tab bar→Reorder, body→Detach, empty→Detach).
