# Wire BufferPool into WindowGpuState for GPU buffer reuse

## Parent Initiative

[[CRT-I-0031]] â€” Performance and Memory Optimization

## Objective

Wire the existing `BufferPool` and `TexturePool` implementations (`src/gpu/`) into `WindowGpuState` to reuse GPU buffers instead of re-allocating them. Currently these pools are fully implemented with RAII patterns and statistics tracking but are not used. Saves ~2MB of re-allocation per window lifecycle.

## Acceptance Criteria

## Acceptance Criteria

## Acceptance Criteria

- [ ] `WindowGpuState` holds a `BufferPool` instance
- [ ] Grid instance buffer (~1.5MB: 32K instances x 48 bytes) allocated from pool
- [ ] Rect instance buffer (~512KB: 16K instances x 32 bytes) allocated from pool
- [ ] Small uniform buffers (256 bytes) allocated from pool
- [ ] Buffers returned to pool on window close/resize (RAII via `PooledBuffer` drop)
- [ ] Pool statistics logged at debug level for monitoring
- [ ] Memory RSS measurement shows reduction during window create/resize/close cycles
- [ ] All existing tests pass

## Implementation Notes

### Technical Approach
1. Add `BufferPool` field to `WindowGpuState`
2. Replace direct `device.create_buffer()` calls with `pool.checkout()` for the three buffer types
3. The existing `PooledBuffer` RAII wrapper handles return-to-pool on drop
4. On resize, return old buffers and checkout new ones at the new size
5. Log pool statistics (hit rate, total allocations) periodically

### Dependencies
None â€” the pool implementation already exists in `src/gpu/buffer_pool.rs`. This task just wires it in.

### Risk Considerations
- **Buffer size mismatch:** Pool returns buffers by size class. If resize changes the buffer size, a new allocation is needed. The pool handles this gracefully â€” it's just a miss, not an error.
- **Thread safety:** `BufferPool` is used only on the render thread. No cross-thread concerns.
