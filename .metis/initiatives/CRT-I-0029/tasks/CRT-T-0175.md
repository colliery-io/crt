# Shell test stabilization

## Parent Initiative

[[CRT-I-0029]]

## Objective

Replace timing-based synchronization (`thread::sleep()`) in shell integration tests with event-driven waiting. Current shell tests use 100ms-5000ms sleeps that cause flakiness in CI (especially on slower runners) and slow down the test suite unnecessarily.

## Acceptance Criteria

## Acceptance Criteria

## Acceptance Criteria

- [ ] All `thread::sleep()` calls in shell/PTY tests replaced with poll-based or event-driven waits
- [ ] Test timeouts reduced from 5s to 1-2s maximum
- [ ] Tests pass reliably on both macOS and Linux CI runners
- [ ] Genuinely slow tests marked with `#[ignore]` and CI annotation
- [ ] No increase in test flakiness (ideally decrease)

## Implementation Notes

### Files to modify
- `tests/shell_tests.rs` — Shell integration tests
- `tests/common/mod.rs` — `TerminalTestHarness` shared helper
- `crates/crt-core/src/pty.rs` — PTY spawn test (100ms sleep)

### Approach
1. **Add `wait_for_output` helper** to `TerminalTestHarness` — polls PTY output with short intervals (10ms) up to a timeout, returns as soon as expected content appears
2. **Add `wait_for_condition` helper** — generic poll-based wait that takes a closure predicate
3. **Replace `sleep(100ms)` + `read_available()`** pattern with `wait_for_output("expected_text", timeout)`
4. **Reduce default timeout** from 5s to 2s — if a shell command hasn't produced output in 2s, something is wrong
5. **Mark truly slow tests** (e.g., large output tests) with `#[ignore]` attribute

### Risk
- Some shell behaviors are timing-dependent (prompt rendering, escape sequence processing) — may need small minimum delays (10-50ms) in addition to event-based waits
- Different shells (bash, zsh, sh) have different startup times — the wait helpers must be shell-agnostic
