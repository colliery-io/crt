# Tab bar state and layout tests

## Parent Initiative

[[CRT-I-0029]]

## Objective

Add comprehensive unit tests for the tab bar state machine and layout engine. These are pure data structures with no GPU dependency — highest ROI tests in the project. Currently only 3 tests exist in `state.rs`; `layout.rs` and `mod.rs` have zero tests.

## Acceptance Criteria

## Acceptance Criteria

## Acceptance Criteria

- [ ] 20+ new tests added across tab bar modules
- [ ] State management fully covered: create, delete, reorder, rename, select, activity indicators
- [ ] Layout calculations covered: tab width, overflow, hit testing, bounds at different widths
- [ ] Edit mode covered: start editing, confirm, cancel, cursor movement, character insertion
- [ ] All existing tests still pass

## Implementation Notes

### Files to test
- `crates/crt-renderer/src/tab_bar/state.rs` — Tab state machine (3 existing tests)
- `crates/crt-renderer/src/tab_bar/layout.rs` — Layout calculations (0 tests)
- `crates/crt-renderer/src/tab_bar/mod.rs` — Public API coordination (0 tests)

### Test areas
1. **State management**: `add_tab`, `close_tab`, `select_tab`, `tab_count`, `active_tab_id` — verify correct tab is active after operations, edge cases (close active tab, close last tab, close only tab)
2. **Layout/hit testing**: `hit_test(x, y)` — verify correct tab ID returned for coordinates, close button detection, clicking between tabs, clicking outside tab area
3. **Tab width calculation**: Verify tabs resize when window width changes, overflow behavior with many tabs
4. **Edit mode**: `start_editing`, `confirm_editing`, `cancel_editing`, `edit_insert_char`, `edit_backspace`, `edit_cursor_left/right/home/end`
5. **Content offset**: `content_offset()` returns correct values for terminal content positioning
