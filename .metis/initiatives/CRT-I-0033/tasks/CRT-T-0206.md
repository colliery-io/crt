## Implementation Notes

### Technical Approach
Mirror the URL hover-underline path in `src/window/render.rs` (`detected_urls` /
`hovered_url_index`, `in_hovered_url` at ~L219–239) and the `RenderContext` default at
~L401–402. Prefer minimal duplication; if approach (B) from the initiative (unified link)
is adopted, this collapses into the URL path.

### Dependencies
Depends on CRT-T-0205 (provides `detected_paths` / `hovered_path_index`).

### Risk Considerations
Keep visual styling identical to URLs for consistency; verify against existing render tests
and (on macOS) golden images if a path-hover visual test is added.

## Status Updates **[REQUIRED]**

**Completed** (2026-06-30).

- `src/window/render.rs`: `RenderContext` gains `hovered_path_index: Option<usize>` and
  `detected_paths: &[DetectedPath]`. The per-cell hover block now computes `in_hovered_url`
  and `in_hovered_path` and underlines when either is set — same `DecorationKind::Underline`,
  so paths look identical to URL hovers.
- Refactor: both checks now call the shared `crate::input::is_position_in_span` (made `pub`),
  removing the previously-inlined span math.
- `src/window/mod.rs`: `RenderContext` construction passes `hovered_path_index` /
  `detected_paths` from `InteractionState`; the test/default ctx passes `None` / `&[]`.

URL precedence (no double underline) is enforced upstream in the input layer (path hover is
cleared whenever a URL is hovered), so at most one of the two flags is ever set per cell.

Tests: new `hovered_path_is_underlined` render test (2 underlines when hovered, 0 when not).
Full render suite: 34 passed, 0 failed. `cargo build --bin crt` green.
