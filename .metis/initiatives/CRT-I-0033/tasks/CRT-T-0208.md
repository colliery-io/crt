## Objective **[REQUIRED]**

Extend path detection (CRT-T-0202) to recognize paths containing spaces — a common OS
pattern — reversing the earlier out-of-scope decision. Support both double/single-quoted
paths (`"My Folder/file.txt"`) and shell-style backslash-escaped spaces
(`My\ Folder/file.txt`), with on-disk existence still the final gate.

## Acceptance Criteria **[REQUIRED]**

- [ ] Double-quoted (`"…"`) and single-quoted (`'…'`) paths with spaces are detected;
  `DetectedPath.path` holds the unquoted text and the span covers the quotes.
- [ ] Backslash-escaped spaces (`\ `) in bare tokens are detected; `path` holds the
  unescaped text while the span covers the on-screen token (incl. backslashes).
- [ ] `:line[:col]` suffixes still parse for quoted/escaped paths.
- [ ] Existing (no-space) detection is unchanged — no regressions in the URL/path suites.
- [ ] Click + resolution open the real spaced path (resolution receives the cleaned text).
- [ ] Unit tests cover each form, multiple quoted paths in a line, span correctness, and an
  end-to-end validate against a real file whose name contains a space.

## Implementation Notes

### Technical Approach
Replace the single bare-run regex with an alternation: double-quoted, single-quoted, or a
bare run that also accepts `\ `. In `detect_paths_in_line`, branch on quoted vs bare to
compute the on-screen token (for the span) and the logical path (for resolution): quoted →
strip quotes; bare → trim trailing punctuation then unescape `\ ` → ` `. Existence is still
the false-positive backstop, so liberal detection is safe.

### Dependencies
Builds on CRT-T-0202 (detection) and CRT-T-0203 (validation). No changes needed to open
(`build_open_command` already keeps a spaced `{file}` a single argv element).

### Risk Considerations
Apostrophes in prose/filenames could start a single-quote scan; an unbalanced quote simply
doesn't match, and any bogus span fails the existence check, so there is no clickable false
positive (only minor wasted work).
